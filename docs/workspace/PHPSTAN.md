# PHPStan workflow

PHPStan runs at `level: max` over `src/` and four packages (`semitexa-core`, `semitexa-ssr`, `semitexa-orm`, `semitexa-tenancy`). Custom Semitexa rules are wired in (`Forbidden*`, `ConfigOn*`, `Inject*`, `StaticContainerAccess`, `Discovery*`, `Factory*`, `Module*`).

## Running it

```bash
composer phpstan          # normal: must report 0 errors
composer phpstan:strict   # strict: also fails if any baseline ignore is unmatched / wrong-count
```

Both run inside the `app` Docker container. Local PHP installations differ from the runtime, so always use the composer scripts.

`phpstan:strict` is the same analysis with `reportUnmatchedIgnoredErrors: true`. It catches:

- baseline entries pointing at deleted/moved files (`Path "/var/www/html/..." is neither a directory, nor a file...`),
- baseline entries that no longer match any reported error (`ignore.unmatched`),
- baseline `count:` fields that overshoot the actual occurrence count (`ignore.count`).

You should run `phpstan:strict` before merging anything that touches an analysed package. CI must run it as a hard gate.

## The "no new baseline entries" rule

The baseline (`phpstan-baseline.neon`) is a frozen list of pre-existing technical debt. **Do not grow it.**

- `composer phpstan:baseline` is intentionally disabled. It exits with a message, not a regenerated file.
- The escape hatch — `composer phpstan:baseline-rebuild` — exists only for one situation: you have explicit, written consent on a PR that says "regenerating the baseline is the correct action here", and you have explained why in the PR description. It should be a one-line decision the reviewer can audit, not a habit.
- A single-entry hand-edit of the baseline is sometimes the right answer when an external constraint makes a fix impossible (a new dependency adds a wrong stub, a generated file needs a brittle assertion). Append it manually with a `# justification: …` comment line above the entry. The reviewer's job is to push back if the justification is thin.
- `// @phpstan-ignore-next-line` is **not** an alternative to fixing code. Use it only when you can't move the code (e.g. it's autogenerated) and the diff would be smaller than the baseline entry. The same justification standard applies.

## When you see a new error from PHPStan

The hierarchy of acceptable responses, in order of preference:

1. Fix the root cause: tighten a return type, add a generic, replace `mixed` with a DTO, narrow at the boundary.
2. Coerce at the boundary: if a value enters from an untyped source (Swoole, JSON, Composer metadata), wrap it once in a typed helper instead of asserting at every consumer.
3. Mark intent semantically: `@phpstan-impure` for static methods that read mutable state, `@phpstan-type` aliases for shapes that recur, `@var` narrowing inside a closure body where PHPDoc on the closure isn't honored.
4. As a last resort, justify and document either a new baseline entry or an inline ignore. Reviewers will ask why.

If you find yourself writing your fifth `is_string($x['k'] ?? null)` guard against the same shape, the problem is the shape — define it as a `@phpstan-type` and propagate, then delete the guards. They're how `function.alreadyNarrowedType` and `nullCoalesce.offset` enter the baseline.

## Why the baseline rotted before

A long-running snapshot accumulated three classes of garbage:

- entries pointing at files that had been deleted or moved by ongoing refactors (API extraction, NATS migration, ORM cleanup) — about 5% of the baseline,
- entries whose error message no longer matched anything PHPStan now reports — about 27%,
- `count:` fields that overshot the actual occurrence count when partial fixes landed without a baseline trim — about 2%.

Together, this hid 10 real errors that were leaking past the analysis at the moment the cleanup epic started, including dead branches and a DI-blocking visibility bug. `phpstan:strict` exists so this can't happen quietly again.

## Helper scripts

When the baseline gets out of sync after a real code change, three idempotent scripts under `bin/phpstan/` can be re-applied:

- `bin/phpstan/strip-stale-baseline.php` — drop entries pointing at non-existent paths.
- `bin/phpstan/strip-unmatched.php` — drop entries that PHPStan reports as `ignore.unmatched`.
- `bin/phpstan/sync-counts.php` — adjust stale `count:` values reported as `ignore.count`.

Typical re-alignment after a sweeping refactor:

```bash
docker compose exec -T app ./vendor/bin/phpstan analyse --memory-limit=1G --no-progress \
  -c phpstan-strict.neon --error-format=json > /tmp/strict.json

php bin/phpstan/strip-stale-baseline.php phpstan-baseline.neon /tmp/cleaned.neon
mv /tmp/cleaned.neon phpstan-baseline.neon

php bin/phpstan/strip-unmatched.php phpstan-baseline.neon /tmp/strict.json /tmp/cleaned.neon
mv /tmp/cleaned.neon phpstan-baseline.neon

php bin/phpstan/sync-counts.php phpstan-baseline.neon /tmp/strict.json /tmp/cleaned.neon
mv /tmp/cleaned.neon phpstan-baseline.neon

composer phpstan:strict   # must report 0 errors when this is done
```

They are not a substitute for the rules above. They exist for the rare case where a sweeping refactor lands and the baseline genuinely needs to be re-aligned with reality. Always run `phpstan:strict` afterwards to confirm the result.
